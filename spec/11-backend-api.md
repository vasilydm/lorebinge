## 8. Backend API (контракты)

### 8.1 Postgres RPC-функции (`SECURITY DEFINER`)

> Вызываются через `supabase.rpc(...)`. Все возвращают JSON, проверяют `auth.uid()`. Обновлено по ревизии: добавлена `start_lesson`, `refill_battery` расширена до `refresh_user_state` (батарея + ленивая переоценка стрика), удалены `buy_streak_freeze`/`apply_streak_freeze` (магазин и ручная заморозка убраны, КРИТ-8); ачивки убраны (ВАЖН-16).

|Функция                 |Вход                           |Выход                                                                                                                |Логика (кратко)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|------------------------|-------------------------------|---------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|`start_lesson`          |`lesson_id uuid`               |`{ ok }`                                                                                                             |**Старт раннера урока (ВАЖН-1/ВАЖН-2).** Сбрасывает `errors_count` текущего урока в 0 (чтобы тон маскота считался за один прогон, а не копился сквозь перепрохождения) и **лениво создаёт** `user_course_progress` (`started_at=now()`) при первом входе в курс. Вызывается клиентом при открытии экрана урока, перед первым заданием.|
|`submit_task_attempt`   |`task_id uuid, is_correct bool`|`{ battery, was_correct, is_subscriber }`                                                                            |Записывает attempt. Если подписка активна (`trial`/`active`) — батарея не тратится. Иначе для теста (`requires_check=true`): `battery=0` → `BATTERY_EMPTY` (→ paywall); иначе `battery -= battery_cost_per_answer` (за любой ответ). `is_correct=false` → `errors_count += 1`. Контентные задания батарею не трогают. (Бонус заряда за урок начисляет `complete_lesson`, §A4.)                                                                                                                                                                                                            |
|`complete_lesson`       |`lesson_id uuid`               |`{ unlocked_next_lesson_id, diamonds_awarded, battery_awarded, already_rewarded, new_diamond_balance, errors_count, course_completed, leveled_up, new_knowledge_level }`|Проверяет, что все задания пройдены (§A7). `status=completed`, `errors_count` (число неверных тест-ответов, для маскота), разблокировка следующего урока, обновление `lessons_completed`. **Первое завершение урока** (`already_rewarded=false`): случайные алмазы из взвешенного `lesson_diamond_rewards` (идемпотентно по леджеру `ref_id=lesson_id`, §A6) + фикс `battery_reward_per_lesson` заряда (`min(battery_max, battery+reward)`; не подписчику; §A4). **Перепрохождение** уже `completed`-урока (`already_rewarded=true`): фикс `lesson_replay_diamond_reward` (дефолт 1, §A6/§12.6), **без** заряда, взвешенный рандом не перекатывается. **Квесты (КРИТ-5):** в той же транзакции инкрементит прогресс активных квестов пользователя с `criteria.lessons_completed`, ставит `completed_at` при достижении цели. **Последний урок курса:** курс `completed`, `courses_completed += 1`; каждые `courses_per_level` (дефолт 5) пройденных курсов → `knowledge_level += 1` (`leveled_up=true`, §A4/§12.0). Вызывает `touch_streak`. *(Ачивок нет — убраны из MVP, ВАЖН-16.)*|
|`touch_streak`          |—                              |`{ current_streak, longest_streak }`                                                                                 |**Внутренняя** (вызывается из `complete_lesson` / `refresh_user_state`, **не** напрямую с клиента). Инкремент стрика по дню; **автозаморозка за алмазы** при пропуске (§A5).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|`refresh_user_state`    |—                              |`{ battery, next_refill_at, current_streak, longest_streak, streak_broken, frozen_days }`                            |**Заменяет `refill_battery`.** Один серверный вызов при старте/заходе (КРИТ-8): (1) ленивое восстановление батареи по `battery_refill_minutes` (§A4); (2) ленивая переоценка стрика — автозаморозка за алмазы за пропущенные дни (`streak_freeze_cost_per_day`/день, пока хватает баланса; `frozen_days` = сколько списано), иначе `streak_broken=true`, стрик → 0 (§A5). Так HUD при входе показывает актуальные батарею **и** стрик.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
|`reset_course_progress` |`course_id uuid`               |`{ ok }`                                                                                                             |Удаляет прогресс по курсу.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
|`get_subscription_offer`|—                              |`{ trial_available, trial_days, timer_active, timer_ends_at, timer_expired }`                                        |**Server-authoritative логика предложения (§A14).** Читает `app_config.economy.trial_offer` и `subscriptions`. Ставит `offer_timer_started_at` на первом вызове при включённом таймере. Клиент только отображает.                                                                                                                                                                                                                                                                                                                                                                         |
|`get_home_feed`         |—                              |`{ lores:[…], in_progress:[], popular:[], categories:[] }`                                                           |Агрегирует блоки Home одним запросом. Состав полей (НЕКР-12): `lores[]` — карточка Lores с вычисленными `lessons_count`/`tasks_count` (+ поля градиента/декора, §12.3а); `in_progress[]` — `{ id, title, cover_image_url, lessons_completed, total_lessons }` (**без** `last_lesson_id` — карточка ведёт на `course_path`, §12.3); `popular[]` — `{ id, title, cover_image_url }`; `categories[]` — `{ id, name, slug }`. Все опубликованные курсы. (Блок Grid — отдельная вкладка → `get_grid_feed`.)                                                                                                                                                                                                                                                                                                                                                             |
|`get_grid_feed`         |—                              |`{ items:[{ course_id, cover_image_url, title }] }`                                                                  |Лента вкладки **Grid** (§12.3б). `cover_image_url` = обложка **первого урока** курса (`lessons.cover_image_url` с минимальным `sort_order` — это и есть «первое видео»). Все `is_published`-курсы; порядок — `sort_order`/`popularity_score`.                                                                                                                                                                                                                                                                                                                                                  |
|`get_category_courses`  |`category_id uuid`             |`{ category:{ id, name }, courses:[{ id, title, cover_image_url, lessons_count, tasks_count }] }`                     |Курсы одной категории (§12.3в), только `is_published`. Счётчики — агрегаты, как в `get_home_feed`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
|`get_course_path`       |`course_id uuid`               |`{ course:{ id,title,path_background_color }, lessons:[{id,title,cover_image_url,sort_order,status}] }`              |Сетка уроков со статусами (`completed`/`active`/`locked`) для пользователя. Возвращает обложку каждого урока и цвет фона сетки. **Ленивая инициализация (ВАЖН-1):** при отсутствии строк прогресса статусы вычисляются по правилу «первый по `sort_order` — `active`, остальные `locked`», строки **не** создаются; `user_course_progress` создаётся позже — в `start_lesson` при первом входе в урок.                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|`get_quests`            |—                              |`{ daily:[], monthly:[], exclusive:[] }`                                                                             |Блок F: активные квесты daily/monthly/exclusive со статусом. **Магазина заморозки нет** (убран, КРИТ-8) — заморозка стрика теперь автоматическая за алмазы (§A5).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
|`claim_quest_reward`    |`quest_id uuid`                |`{ diamonds, claimed }`                                                                                              |Начисляет алмазы (идемпотентно по леджеру `quest_reward`/`ref_id`). `QUEST_NOT_COMPLETED` если не выполнено.                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
|`delete_account`        |—                              |`{ ok }`                                                                                                             |Удаляет данные пользователя (каскад) и аккаунт. **Требование и Google Play, и App Store** (Apple 5.1.1(v)).                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

**Транзакционность.** `complete_lesson` — одна транзакция; начисление алмазов за первое прохождение идемпотентно (проверка по леджеру `ref_id`); обновление квестов и счётчика курсов/уровня — в той же транзакции.

> **Доверие клиенту в `submit_task_attempt(is_correct)` — осознанный компромисс (НЕКР-15).** Правильные ответы лежат в `payload` (читаемом клиентом), и клиент сам сообщает `is_correct`. Это формально расходится с принципом §3 «не доверяем клиенту», но безопасно: `is_correct` влияет **только** на `errors_count` (вход для тона маскота, F7); экономика (батарея/алмазы) от правильности **не зависит** и считается сервером. Серверную проверку ответов в MVP не строим.

### 8.2 Edge Functions (TypeScript)

|Функция             |Метод         |Вход                          |Выход                             |Назначение                                                                                                                                                                                                                                                                                                                                                          |
|--------------------|--------------|------------------------------|----------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|`revenuecat-webhook`|POST (webhook)|RevenueCat event payload      |200                               |**Источник истины по подпискам обеих платформ.** Принимает события RevenueCat (initial_purchase, renewal, trial_started, cancellation, expiration, billing_issue) и обновляет `subscriptions` + пишет `payment_events` (с `platform`/`store`, суммой, валютой, страной). На событии `trial_started` дополнительно ставит `subscriptions.trial_used = true` (§A14). Проверка подписи вебхука. Заменяет прежний `play-rtdn` и покрывает App Store + Google Play.|
|`validate-purchase` |POST          |`{ rc_app_user_id }` (или чек)|`{ status, expires_at, platform }`|Реконсиляция статуса по запросу клиента после покупки/восстановления: сверяется с RevenueCat REST API, обновляет `subscriptions`. (Fallback без RevenueCat: валидация чека Apple через App Store Server API **или** Google Play Developer API по `platform`.) Ключи — в секретах Edge Functions.                                                                    |


> **Поток подписки (RN).** Клиент покупает/восстанавливает через RevenueCat SDK (`react-native-purchases`) → RevenueCat обновляет entitlement → шлёт `revenuecat-webhook` → Edge Function пишет `subscriptions`. Клиент дополнительно может дернуть `validate-purchase` для немедленной реконсиляции. Статус в приложении читается из `subscriptions` (server-authoritative), не из локального кэша SDK.
