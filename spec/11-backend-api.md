## 8. Backend API (контракты)

### 8.1 Postgres RPC-функции (`SECURITY DEFINER`)

> Вызываются через `supabase.rpc(...)`. Все возвращают JSON, проверяют `auth.uid()`. Обновлено по ревизии: добавлена `start_lesson` (с возобновлением прогресса урока, §A7), `refill_battery` расширена до `refresh_user_state` (батарея + ленивая переоценка стрика), удалены `buy_streak_freeze`/`apply_streak_freeze` (магазин и ручная заморозка убраны, КРИТ-8); ачивки убраны (ВАЖН-16); **квесты убраны целиком** — `get_quests`/`claim_quest_reward` удалены (глобальное решение).

|Функция                 |Вход                           |Выход                                                                                                                |Логика (кратко)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|------------------------|-------------------------------|---------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|`start_lesson`          |`lesson_id uuid`               |`{ ok, resume_task_index }`                                                                                          |**Старт раннера урока (прогресс сохраняется, §A7).** Лениво создаёт `user_course_progress` (`started_at=now()`) и `user_lesson_progress` (`status=active`) при первом входе. **Возобновление:** возвращает `resume_task_index = current_task_index` — раннер открывается с этого задания (не с начала). Если урок уже `completed` (перепрохождение) → `current_task_index` и `errors_count` сбрасываются в 0, `resume_task_index=0` (проходим заново). Если урок `active` и `current_task_index=0` (первый заход) → `errors_count=0`. Если `active` и `current_task_index>0` (продолжение после выхода/окончания заряда) → `errors_count` и индекс **сохраняются**. Вызывается клиентом при открытии экрана урока.|
|`submit_task_attempt`   |`task_id uuid, is_correct bool`|`{ battery, was_correct, is_subscriber }`                                                                            |Записывает attempt и **продвигает `user_lesson_progress.current_task_index`** на следующее задание (по `sort_order`) — прогресс урока сохраняется. Если подписка активна (`trial`/`active`) — батарея не тратится. Иначе для теста (`requires_check=true`): `battery=0` → `BATTERY_EMPTY` (→ paywall; индекс **не** двигается, чтобы при возврате продолжить с этого же задания, §12.9/Gap #3). Иначе `battery -= battery_cost_per_answer` (за любой ответ). `is_correct=false` → `errors_count += 1`. Контентные задания (`requires_check=false`) батарею не трогают, но индекс продвигают.                                                                                                                                                                                                            |
|`complete_lesson`       |`lesson_id uuid`               |`{ unlocked_next_lesson_id, diamonds_awarded, already_rewarded, new_diamond_balance, errors_count, course_completed, leveled_up, new_knowledge_level }`|Проверяет, что все задания пройдены (§A7). `status=completed`, `errors_count` (число неверных тест-ответов, для маскота), разблокировка следующего урока, обновление `lessons_completed`. **Награда за урок — только алмазы** (заряд за завершение не начисляется, §A4/§12.6). **Первое завершение урока** (`already_rewarded=false`): случайные алмазы из взвешенного `lesson_diamond_rewards` (идемпотентно по леджеру `ref_id=lesson_id`, §A6). **Перепрохождение** уже `completed`-урока (`already_rewarded=true`): фикс `lesson_replay_diamond_reward` (дефолт 1, §A6/§12.6), взвешенный рандом не перекатывается. **Последний урок курса:** курс `completed`, `courses_completed += 1`; каждые `courses_per_level` (дефолт 5) пройденных курсов → `knowledge_level += 1` (`leveled_up=true`, §A4/§12.0). Вызывает `touch_streak`. *(Ачивок и квестов нет — убраны из MVP.)*|
|`touch_streak`          |—                              |`{ current_streak, longest_streak }`                                                                                 |**Внутренняя** (вызывается из `complete_lesson` / `refresh_user_state`, **не** напрямую с клиента). Инкремент стрика по дню; **автозаморозка за алмазы** при пропуске (§A5).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|`refresh_user_state`    |—                              |`{ battery, next_refill_at, current_streak, longest_streak, streak_broken, frozen_days }`                            |**Заменяет `refill_battery`.** Один серверный вызов при старте/заходе (КРИТ-8): (1) ленивое восстановление батареи по `battery_refill_minutes` (§A4); (2) ленивая переоценка стрика — автозаморозка за алмазы за пропущенные дни (`streak_freeze_cost_per_day`/день, пока хватает баланса; `frozen_days` = сколько списано), иначе `streak_broken=true`, стрик → 0 (§A5). Так HUD при входе показывает актуальные батарею **и** стрик.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
|`reset_course_progress` |`course_id uuid`               |`{ ok }`                                                                                                             |Удаляет прогресс по курсу.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
|`get_subscription_offer`|—                              |`{ trial_available, trial_days, timer_active, timer_ends_at, timer_expired }`                                        |**Server-authoritative логика предложения (§A14).** Читает `app_config.economy.trial_offer` и `subscriptions`. Ставит `offer_timer_started_at` на первом вызове при включённом таймере. Клиент только отображает.                                                                                                                                                                                                                                                                                                                                                                         |
|`get_home_feed`         |—                              |`{ lores:[…], in_progress:[], popular:[] }`                                                                          |Агрегирует блоки Home одним запросом. Состав полей (НЕКР-12): `lores[]` — карточка Lores с вычисленными `lessons_count`/`tasks_count` (+ поля градиента/декора, §12.3а) **+ `is_completed`** (курс пройден пользователем — `user_course_progress.completed_at != null`; для overlay «пройдено», §12.3а/D); `in_progress[]` — `{ id, title, cover_image_url, lessons_completed, total_lessons }` (**без** `last_lesson_id` — карточка ведёт на `course_path`, §12.3; по определению не завершён); `popular[]` — `{ id, title, cover_image_url, is_completed }`. Все опубликованные курсы. **Категории Home больше не возвращает** (блок Categories убран с Home; их отдаёт `get_catalog` для вкладки Catalog, §12.3г). (Блок Grid — отдельная вкладка → `get_grid_feed`.)                                                                                                                                                                                                                                                                                                                                                             |
|`get_grid_feed`         |—                              |`{ items:[{ course_id, cover_image_url, title, is_completed }] }`                                                    |Лента вкладки **Grid** (§12.3б). `cover_image_url` = обложка **первого урока** курса (`lessons.cover_image_url` с минимальным `sort_order` — это и есть «первое видео»). `is_completed` — курс пройден пользователем (для тёмного overlay + значка «пройдено», §12.3б/D). Все `is_published`-курсы; порядок — `sort_order`/`popularity_score`.                                                                                                                                                                                                                                                                                                                                                  |
|`get_category_courses`  |`category_id uuid`             |`{ category:{ id, name }, courses:[{ id, title, cover_image_url, lessons_count, tasks_count, is_completed }] }`       |Курсы одной категории (§12.3в), только `is_published`. Счётчики — агрегаты, как в `get_home_feed`; `is_completed` — для overlay «пройдено» (§12.3в/D).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
|`get_catalog`           |—                              |`{ categories:[{ id, name, slug, courses_count }] }`                                                                 |**Алфавитный индекс категорий для вкладки Catalog (§12.3г).** Только категории с **≥1 `is_published`-курсом** (пустые не возвращаются); `courses_count` — число опубликованных курсов в категории. Сортировка — по `name` (алфавит, `localeCompare`; `categories.sort_order` не используется). Группировку по буквам делает клиент. Вкладка по умолчанию выключена (`enabled:false`), но RPC работает всегда — данные появятся, как только админ включит Catalog (B6.7).                                                                                                                                                                                                                                                       |
|`get_course_path`       |`course_id uuid`               |`{ course:{ id,title,path_background_color,is_completed }, lessons:[{id,title,cover_image_url,sort_order,status}] }`  |Сетка уроков со статусами (`completed`/`active`/`locked`) для пользователя. Возвращает обложку каждого урока, цвет фона сетки и **`course.is_completed`** (для верхней плашки «Course completed», §12.4/D). **Ленивая инициализация:** при отсутствии строк прогресса статусы вычисляются по правилу «первый по `sort_order` — `active`, остальные `locked`», строки **не** создаются; `user_course_progress`/`user_lesson_progress` создаются позже — в `start_lesson` при первом входе в урок (§A7).                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|`delete_account`        |—                              |`{ ok }`                                                                                                             |Удаляет данные пользователя (каскад) и аккаунт. **Требование и Google Play, и App Store** (Apple 5.1.1(v)).                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

**Транзакционность.** `complete_lesson` — одна транзакция; начисление алмазов за первое прохождение идемпотентно (проверка по леджеру `ref_id`); счётчик курсов/уровня обновляется в той же транзакции. *(Квесты убраны — RPC `get_quests`/`claim_quest_reward` удалены, обновления прогресса квестов в `complete_lesson` больше нет.)*

> **Доверие клиенту в `submit_task_attempt(is_correct)` — осознанный компромисс (НЕКР-15).** Правильные ответы лежат в `payload` (читаемом клиентом), и клиент сам сообщает `is_correct`. Это формально расходится с принципом §3 «не доверяем клиенту», но безопасно: `is_correct` влияет **только** на `errors_count` (вход для тона маскота, F7); экономика (батарея/алмазы) от правильности **не зависит** и считается сервером. Серверную проверку ответов в MVP не строим.

### 8.2 Edge Functions (TypeScript)

|Функция             |Метод         |Вход                          |Выход                             |Назначение                                                                                                                                                                                                                                                                                                                                                          |
|--------------------|--------------|------------------------------|----------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|`revenuecat-webhook`|POST (webhook)|RevenueCat event payload      |200                               |**Источник истины по подпискам обеих платформ.** Принимает события RevenueCat (initial_purchase, renewal, trial_started, cancellation, expiration, billing_issue) и обновляет `subscriptions` + пишет `payment_events` (с `platform`/`store`, суммой, валютой, страной). На событии `trial_started` дополнительно ставит `subscriptions.trial_used = true` (§A14). Проверка подписи вебхука. Заменяет прежний `play-rtdn` и покрывает App Store + Google Play.|
|`validate-purchase` |POST          |`{ rc_app_user_id }` (или чек)|`{ status, expires_at, platform }`|Реконсиляция статуса по запросу клиента после покупки/восстановления: сверяется с RevenueCat REST API, обновляет `subscriptions`. (Fallback без RevenueCat: валидация чека Apple через App Store Server API **или** Google Play Developer API по `platform`.) Ключи — в секретах Edge Functions.                                                                    |


> **Поток подписки (RN).** Клиент покупает/восстанавливает через RevenueCat SDK (`react-native-purchases`) → RevenueCat обновляет entitlement → шлёт `revenuecat-webhook` → Edge Function пишет `subscriptions`. Клиент дополнительно может дернуть `validate-purchase` для немедленной реконсиляции. Статус в приложении читается из `subscriptions` (server-authoritative), не из локального кэша SDK.
