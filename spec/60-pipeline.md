# Lorebinge — пайплайн и иерархия промптов (v3)

> Рабочий документ. Весь путь: списки идей/топиков → итеративные апрувы → канон →
> разбиение большого промпта на генераторы B1–B5 → апрув персонажей (2 шага) → апрув
> промптов → производство медиа → поэлементный апрув → импорт. Заменяет прошлую версию.

---

## 0. Две базовые таблицы-списка (основание)

- **Список идей** — короткие зёрна из 2–3 слов. Ещё не курс; из идеи позже может родиться
  топик.
- **Список топиков** — полноценные предложения, которые могут лечь в основу курса. К топику
  можно приложить «кусок сценария» — дополнение к промпту (= `USER_BRIEF`, доходит до B2).

---

## 1–3. Ручной ввод из админки и приоритет очереди

1. **Ручной ввод идеи** (вне апрува): идею одним полем → в очередь как идея (не топик).
2. **Ручной ввод топика** (вне апрува): `топик` + `кусок сценария` → пара в очередь.
3. **Приоритет очереди:** сначала готовые топики; когда кончаются — апрув идей.

---

## A. Механика апрува (ОДНА на все шлюзы ⛔)

Каждый шлюз (идея, топик, канон, фото, промпты, элементы) работает одинаково:

- **✓ Approve** — дальше (где уместно — опц. поле «дописать к промпту»).
- **✗ Reject** — артефакт отбрасывается (для идеи/топика — гибель ветки).
- **✎ Refine** — поле для комментария → **универсальный промпт-переписыватель** берёт
  `{тип артефакта, текущая версия (JSON), твоё замечание, правила этапа}` и возвращает
  **чистую новую версию** (не дописку в конец!), не нарушая жёстких правил этапа → снова
  на апрув. **Цикл повторяется сколько угодно раз, пока не устроит.**

**Почему переписывание, а не дописка:** дописка в конец копит противоречия («Николаю 20» +
снизу «нет, 3») и ломает модель. Переписыватель отдаёт цельный непротиворечивый промпт.
Прозрачность не теряется, потому что:

> **Все версии артефактов + твои замечания хранятся в БД** (история правок) — чистый промпт
> уходит в ИИ, а аудит-трейл остаётся. Это же потом — материал для доработки самого пайплайна.

Переписыватель — ОДИН генерик на все этапы (не серия под каждый), различается только
входным `тип артефакта` + `правила этапа`. Файл: `refine_rewriter.md`.

## 4. Основной процесс (воронка с итеративными апрувами)

```
P1  Генератор идей ──► список идей
                         │ (по одной)
                         ▼
                  ⛔ АПРУВ ИДЕИ            (Approve / Reject)
                         │ ✓
                         ▼
P2  Идея → Топик
                         ▼
                  ⛔ АПРУВ ТОПИКА ⟳        (Reject / Refine+поле / Approve+опц. дописка)
                         │ ✓
                         ▼
P3  Генератор СЦЕНАРИЯ-КУРСА (учитывает дописку/кусок сценария) → КАНОН
     · канон включает РОСТЕР персонажей и их количество
                         ▼
                  ⛔ АПРУВ КАНОНА ⟳        (правлю всё: число уроков, число персонажей,
                         │ ✓                убрать конкретного — любым замечанием; персонажи
                         │                  одобряются ЗДЕСЬ, отдельного апрува списка нет)
                         ▼
            B5 refs → фото персонажей (1 на каждого) ─┐
            B6      → стилевые фото (сколько влезет в omni) ─┤
                         ▼                                   │
                  ⛔ АПРУВ ФОТО (персонажи + стиль) ⟳ ───────┘  (визуальный фундамент,
                         │ ✓                                     ДО видео — дёшево чинить)
                         ▼
            B1/B2/B3/B4 заполняют слоты → ПРОМПТЫ
                         ▼
                  ⛔ АПРУВ ПРОМПТОВ ⟳      (каждый промпт + RU-перевод + RU-пересказ; §8.5)
                         │ ✓
                         ▼
            Производство медиа (omni/Leonardo/ElevenLabs)  (§9)
              ├─ видео 1 готово → ⛔ АПРУВ ВИДЕО 1 ⟳  (поштучно, §11)
              ├─ видео 2 готово → ⛔ АПРУВ ВИДЕО 2 ⟳
              └─ видео 3 готово → ⛔ АПРУВ ВИДЕО 3 ⟳
                         ▼
                  ⛔ АПРУВ ОСТАЛЬНОГО ⟳    (батч: все картинки курса + все тексты; §12)
                         │ ✓
                         ▼
                  Импорт в public → приложение + маркетинг
                         ▲
                     P4  Доработка одного элемента (по Refine-комментарию)
```

`⟳` = итеративная петля апрува (см. §A): правлю → возвращается → пока не устроит.

## 5. Что выдаёт P3 (канон курса) и куда уходит каждый слот

P3 НЕ пишет финальные промпты — задаёт КАНОН и слоты, которые заполняют B-генераторы.
Канон = `{ course_facts, narrative, lesson_plan, 3 видео-акта, character roster,
edge_element, hook }`.

Слоты курса (звёздочкой* — достроено после обрыва заметки):

1. Промпт **главной обложки** (центр карточки) ──────────► **B3** `course_illustration`
2. Промпт **мини-картинки** (низ карточки) ─────────────► **B4** `card_decoration`
3. **3 видео** (видео-уроки = 3 акта) ──────────────────► **B2** (каждое → 6 kling-промптов)
4. *Обложки уроков* ────────────────────────────────────► **B3** `lesson_cover`
5. *Текст + задания обычных уроков* ────────────────────► **B1** (+ image-брифы → B3)
6. *Картинки в уроках (reading_media, image_recognition)* ► **B3**
7. *Ростер персонажей (имена, число)* ──────────────────► **в каноне** (одобряется при апруве канона)

> Канон полностью редактируем на апруве (§6/этап 5): число уроков, число персонажей, убрать
> конкретного — любым замечанием, сценарий переписывается. БД принимает переменное число и
> уроков, и персонажей (массивы `lessons[]`/`characters[]` любой длины).
> ⚠️ Пункты 4–6 достроены по уже зафиксированному — заметка оборвалась на «3 промпта для…».

## 6. Иерархия промптов (верх → низ)

```
ВЕРХ — ВОРОНКА И КАНОН
├── P1  Генератор идей            — 2–3-словные зёрна, без повторов
├── P2  Идея → Топик             — развернуть апрувнутую идею в топик
└── P3  Генератор сценария-курса — КАНОН для всех B (факты, нарратив, план уроков,
                                    3 видео-акта, ростер персонажей, edge/хук)

НИЗ — ГЕНЕРАТОРЫ АРТЕФАКТОВ (бывший монолит P3; каждый отдельный, все едят канон)
├── B1  Текст + задания  — reading_text/reading_media + quiz-типы               [ГОТОВ]
├── B2  Видео (omni)     — 3 видео × 6 kling_prompt → omni-формат (image_list)   [ОБНОВИТЬ]
├── B3  Большие картинки — illustration / lesson_cover / reading_media / image  [ГОТОВ]
├── B4  Угловой декор    — card_decoration: угловой прозрачный PNG (Leonardo)     [ГОТОВ]
├── B5  Фото персонажей  — refs: character_lock + 1 фото на персонажа (ростер уже в каноне)  [ГОТОВ]
└── B6  Стилевые фото    — набор образцов стиля без персонажей (апрув)           [TODO]

P4  Доработка — точечная перегенерация ОДНОГО элемента по Refine-комментарию.
```

**Канон-контракт на вход каждому B:** `{ course_facts, narrative, lesson_plan, characters,
edge_element, hook }`. «Раздельно» ≠ «вразнобой» — все читают один канон.

## 7. Порядок продакшена и апрув визуального фундамента

Ростер персонажей уже одобрен в каноне (§6). Дальше — фото. B2 (omni-видео) и B3 ссылаются
на залоченные референс-фото → их готовим и апрувим ПЕРЕД видео/картинками.

```
1. B5 refs → фото персонажей (1 на каждого из ростера канона) ─┐
2. B6      → стилевые фото (без персонажей)                     ─┤→ ⛔ АПРУВ ФОТО ⟳
   ─────────────────────────────────────────── только потом:    ┘   (персонажи + стиль вместе)
3. B2 omni-видео  (в image_list: фото персонажей + стиль-фото; бюджет ≤ 7, см. §9)
4. B3 визуал      (нужны фото-референсы персонажей)
5. B4 декор       (персонажи не нужны — параллельно)
```

- Отдельного апрува *списка* персонажей нет — список решается в каноне. Апрувится только
  визуальный результат (фото). Один объединённый шлюз: фото персонажей + стилевые фото.
- Зачем шлюз ДО видео: чинить кривое фото дёшево, перегенерировать все клипы — дорого.
- Стилевых фото апрувим столько, сколько влезает в omni: потолок вызова = 7 картинок всего,
  персонажи в приоритете, стилю — остаток (чуть меньше).
- Библиотечные совпадения персонажей по aliases берём как есть — повторно не апрувим.
- Стиль фото персонажа = стиль видео (реально-мультяшный) — задаёт вид во всём медиа.

## 8. Анатомия курса

- **Курс** = 6–10 уроков. **Видео-урок** — только видео, позиции **1, 4, 7** (3 акта).
  **Обычный урок** — 6–8 заданий: сначала ВЕСЬ текст, потом ВСЕ quiz.
- Видео двойного назначения: урок в приложении + маркетинг в соцсетях.
- Пример (9): `1-видео·2·3·4-видео·5·6·7-видео·8·9` → 3 видео + 6 обычных.
- Типы заданий §6.5; `requires_check`: чтение — Continue, quiz — проверка.

## 8.5. Апрув промптов (перед производством медиа)

После того как B-генераторы выдали все промпты, но ДО дорогой генерации медиа — отдельный
апрув промптов. Каждый промпт показывается с:

- **подробным переводом на русский**,
- **кратким пересказом на русском** (чтобы не вчитываться — быстрый взгляд),
- возможностью развернуть удобный тебе формат и оценить, что сгенерилось.

Refine-комментарий → перегенерённый промпт → снова апрув (цикл §A). Смысл шлюза: кривой
промпт правим дёшево текстом, а не пережигаем кредиты Kling/Leonardo/ElevenLabs.

## 9. Производство медиа через Kling Omni API

Видео генерится **omni-моделью** Kling: `POST /v1/videos/omni-video`, модель
`kling-v3-omni` (или `kling-video-o1`). Omni принимает в `image_list` РАЗНЫЕ референс-фото —
и персонажей, и **стиля** — адресуемые в промпте через `<<<image_N>>>`.

**Параметры вызова (наши дефолты):** `duration: "15"`, `mode: "pro"` (1080p),
`aspect_ratio: "9:16"`, `sound: "off"` (озвучку кладём через ElevenLabs).
`multi_prompt` (до 6 раскадровок, суммарно = duration) → сюда ложатся наши нарезки 3–4с.

**Как задаётся стиль (главное новое):** в `image_list` кладём фото-образцы стиля и в
промпте пишем «The video style matches that of `<<<image_k>>>`». Это и есть передача
дизайна примером-картинкой (а не текстом «образца-ролика» — такого входа у Kling нет).

**Бюджет референс-фото в ОДНОМ вызове omni:**
- Жёсткий максимум `image_list` = **7** фото (путь без reference-видео, только image-референсы).
- Если использовать «видео-элементы» персонажей из библиотеки (element_list) — лимит падает
  до 4. Поэтому идём через `image_list` ради бюджета 7.
- **Раскладка бюджета:** фото персонажей — приоритет (по 1 на каждого персонажа в этом видео);
  стилевые фото забирают остаток. То есть `#персонажей + #стиль (+1 если якорь-first_frame) ≤ 7`.
  Стилю выделяем чуть меньше: при 3 персонажах → до 4 стиль-фото; при 4 → до 3; и т.д.
- ⚠️ `kling-video-o1` не поддерживает first/end frame, если в массиве >2 картинок. Значит
  при петле через якорь-кадр + несколько референсов берём `kling-v3-omni` (до 3 элементов с
  first_frame) ИЛИ держим петлю промптом/стилем, а не пиннингом first_frame. — **решение ниже.**

После генерации: ffmpeg склеивает 2×15с в три 30-сек ролика → Cloudflare Stream;
ElevenLabs озвучивает по спикерам → ручная синхронизация рта; Leonardo отдаёт картинки в Storage.

**Апрув медиа делится на два:**
- **Видео — поштучно (этап 11):** как только готов ролик 1 — он сразу летит тебе на проверку;
  готов 2 — проверка; готов 3 — проверка. Не ждём всю тройку. (`review_items.kind=video_clip`,
  по одному на видео; Refine → перегенерация только этого ролика.)
- **Всё остальное — батчем (этап 12):** все картинки курса (обложка, мини-картинка, обложки
  уроков, картинки в уроках) + все тексты и задания апрувятся вместе, отдельно от видео.

## 10. Разделение труда: драма против точности

- **Видео (B2)** — драма ×10, точность отключена, comment-bait, soap-opera крючки (+ маркетинг).
- **Текст (B1) и картинки (B3)** — якорь точности: факты, даты, костюм, эпоха верны.

# ЧАСТЬ II — Инфраструктура и реализация

> Добавлено к архитектуре промптов выше. Адаптировано под наши решения: omni-видео,
> B-split, апрувер внутри админки, связь Mac mini ↔ админка **только через Supabase**.
> Tailscale / Jinja-HTMX / туннели НЕ используются.

## 12. Как живёт конвейер (архитектура и связь)

- **Конвейер** — Python-воркер на **Mac mini** (APScheduler + asyncio), держится живым через
  launchd, без сна (pmset). Генерирует контент и медиа, пишет состояние в Supabase.
- **Апрувер** — НЕ отдельное приложение и НЕ на Mac mini. Это **раздел админки** (Приложение B):
  React + shadcn/ui + Tailwind, та же палитра (§5 спеки), адаптив под телефон. Хостинг как у
  админки (Vercel/Netlify), вход по ссылке с авторизацией.
- **Связь — только через Supabase.** Mac mini-во드keр читает/пишет схему `pipeline.*`
  сервисным ключом (локально). Админка читает/пишет `pipeline.*` через **Edge Function**
  (сервисный ключ в браузер не попадает) либо admin-RLS. Прямого соединения админка↔Mac mini
  и туннеля НЕТ — они встречаются в БД.
- **Поток управления:** воркер генерит → пишет в `pipeline.*` → админка показывает на апрув →
  я апрувлю/правлю (запись обратно в `pipeline.*`) → воркер видит при поллинге → двигает стадию
  дальше / запускает доработку. email — только уведомления-ссылки.

```
[Mac mini: Python-воркер]  --service key-->  [ Supabase: схема pipeline.* ]  <--Edge Fn--  [Админка (React/shadcn): апрувер]
   APScheduler + поллер                          (единственная точка встречи)                 /queue /ideas /topic /canon /photos /prompts /videos /content
   Claude/omni/Leonardo/ElevenLabs                                                            email-уведомления со ссылкой
```

## 13. SQL-схема `pipeline` и общение с БД

Отдельная схема `pipeline` в той же Supabase (RLS: сервисная роль для воркера; админка — через
Edge Function / admin-RLS). В `public` импортируется только готовый курс (идемпотентно по `slug`).

```sql
create schema if not exists pipeline;

-- Проект курса: движется по стадиям нашей воронки
create table pipeline.projects (
  id            uuid primary key default gen_random_uuid(),
  stage         text not null default 'idea_pool',
    -- idea_pool|idea_approved|topic_draft|topic_approved|canon_draft|canon_approved
    -- |photos_pending|photos_approved|prompts_pending|prompts_approved
    -- |in_production|published|rejected
  media_status  text,
    -- video1_pending|video1_approved|video2_pending|video2_approved
    -- |video3_pending|video3_approved|content_pending|content_approved|ready_to_mux|done
  slug          text,
  working_title text,
  category      text,
  user_note     text,            -- дописка при апруве топика = USER_BRIEF (доходит до B2)
  idea_json     jsonb,           -- стадия 1 (P1)
  topic_json    jsonb,           -- идея→топик (P2)
  canon_json    jsonb,           -- канон (P3): факты, нарратив, план уроков, 3 видео-акта,
                                 --             ростер персонажей, edge_element, hook
  course_json   jsonb,           -- собранный курс (B1–B4 заполнили слоты) → импорт в public
  media_json    jsonb,           -- ссылки на видео/аудио/картинки + статусы
  imported_course_id uuid,       -- public.courses.id после импорта
  created_at    timestamptz default now(),
  updated_at    timestamptz default now()
);
-- Отличия от старой спеки: нет scenarios_json/chosen_scenario (3 варианта не делаем);
-- добавлен topic_json (шаг идея→топик); scenario→canon переименован.

-- Глобальная библиотека персонажей (переиспользуется между курсами)
create table pipeline.characters (
  id             uuid primary key default gen_random_uuid(),
  canonical_name text not null unique,
  aliases        text[] not null,
  is_historical  bool default true,
  character_lock text,                 -- текстовый дескриптор (бэкап)
  photo_urls     text[] not null,      -- фото-референс(ы); для omni берём 1 в image_list
  created_at     timestamptz default now()
);
-- kling_subject_id убран: omni берёт фото по URL в image_list, Subject Library не используется.

create table pipeline.project_characters (
  project_id   uuid references pipeline.projects(id),
  character_id uuid references pipeline.characters(id),
  primary key (project_id, character_id)
);

-- Стилевые фото проекта (B6) — образцы дизайна без персонажей, для omni image_list
create table pipeline.style_photos (
  id         uuid primary key default gen_random_uuid(),
  project_id uuid references pipeline.projects(id),
  url        text not null,
  facet      text,                     -- lighting|palette|texture|composition
  approved   bool default false,
  created_at timestamptz default now()
);

-- Свои темы вручную (приоритетные семена)
create table pipeline.seed_topics (
  id         uuid primary key default gen_random_uuid(),
  title      text not null,
  premise    text,
  user_note  text,                     -- кусок сценария / дописка к промпту
  mode       text default 'to_canon',  -- to_canon (готовый топик) | to_idea_batch
  priority   bool default true,
  status     text default 'pending',   -- pending | consumed
  created_at timestamptz default now()
);

-- Поэлементный апрув: каждый проверяемый элемент — строка
create table pipeline.review_items (
  id             uuid primary key default gen_random_uuid(),
  project_id     uuid references pipeline.projects(id),
  kind           text not null,
    -- idea|topic|canon|character_photo|style_photo
    -- |prompt (любой B-промпт, с RU-переводом+пересказом)
    -- |video_clip (по одному на видео)
    -- |reading_text|reading_media_text|matching|fill_blank|image_recognition
    -- |multiple_choice|true_false|timeline
    -- |course_meta|cover_image|illustration|card_decoration|reading_media_image|image_recognition_image|voiceover
    -- (НЕТ retelling, НЕТ auto_check, НЕТ is_feed — сознательно, см. §18)
  locator        jsonb not null,       -- путь внутрь canon/course_json
  content        jsonb,                -- текущее значение
  content_ru     jsonb,                -- для kind=prompt: RU-перевод + краткий RU-пересказ (Этап 10)
  status         text default 'pending', -- pending|approved|needs_revision|regenerating
  comment        text,                 -- моё замечание (Refine)
  revision_count int default 0,
  version_history jsonb,               -- все версии артефакта + замечания (прозрачность варианта 2)
  created_at     timestamptz default now(),
  updated_at     timestamptz default now()
);

create table pipeline.events (
  id uuid primary key default gen_random_uuid(),
  project_id uuid references pipeline.projects(id),
  type text not null, payload jsonb, created_at timestamptz default now()
);
```

**Общение с БД:** воркер — `supabase-py` сервисным ключом; админка — через Edge Function
`pipeline-approval` (валидация + запись статуса/замечания) и чтение `pipeline.*`. Импорт в
`public` — `importer.py`, идемпотентно по `slug`.

## 15. Веб-страницы апрува (в админке) + UX

Все — разделы админки (React + shadcn/ui), адаптив веб+телефон. На каждой — механика §A
(Approve / Reject / Refine), плюс анти-случайный-тап: Approve заблокирован без галочки
«I reviewed»; Reject — без галочки + непустого комментария; необратимое («в продакшен»,
импорт) — двойное подтверждение; валидация на сервере (Edge Function).

| Маршрут        | Шлюз (этап)                         | Что показывает / UX |
|----------------|-------------------------------------|---------------------|
| `/queue`       | витрина всех проектов               | карточки: прогресс-бар, степпер, бейдж, флаг «ждёт меня» наверх; CTA на нужный экран |
| `/ideas`       | апрув идей (эт. 2)                   | список идей-зёрен, Approve/Reject |
| `/topic`       | апрув топика (эт. 4)                 | топик; Reject / Refine+поле / Approve+опц. дописка (USER_BRIEF) |
| `/canon`       | апрув канона (эт. 5/6)              | весь канон: план уроков, **ростер персонажей с числом** (правлю любым замечанием — число уроков/персонажей/убрать конкретного); персонажи одобряются здесь |
| `/photos`      | апрув фото (эт. 7+8)                | фото персонажей + стилевые фото вместе; до видео |
| `/prompts`     | апрув промптов (эт. 10)             | каждый промпт + **RU-перевод + краткий RU-пересказ**, разворот удобного формата; Refine |

## 16. Надёжность

- **Ретраи:** `httpx` с backoff. Кривой JSON от LLM → авто-повтор «return valid JSON only».
- **Деградация:** падение omni/Leonardo/ElevenLabs → проект застывает на текущем `media_status`,
  email-алерт, повтор кнопкой в админке.
- **Kill-switch:** `PIPELINE_ENABLED=false` (env) — стоп всей автогенерации одним переключателем.
- **Идемпотентность:** импорт по `slug`; антидубли идей (`avoid_list` в P1); антидубли персонажей
  (поиск по aliases). Каждый шаг пишет `pipeline.events`.
- **Версионирование:** все версии артефактов + замечания в `review_items.version_history`
  (прозрачность переписывателя, §A).
- **Heartbeat:** email раз в сутки «конвейер жив».
- **Секреты:** только в `.env` на Mac mini; сервисный ключ — только в воркере, никогда в браузере.

## 17. Бюджет и доступ к Kling (fal.ai)

- **Доступ к Kling — через fal.ai (PAYG).** НЕ прямой Kling API: для одиночки прямой API —
  стартовый пакет ~$4200 за ~30k единиц + ~3 мес обязательств. fal.ai — pay-as-you-go без
  предоплаты, тот же ключ уже под omni и (опц.) под картинки (Flux).
- **Математика видео (порядок величин, omni pro ~7 кредитов/сек или ~$0.05/сек — проверить
  актуальный тариф omni на fal.ai):**
  - 1 ролик = 2×15с = 30с генерации → ~210 кредитов / ~$1.5.
  - 1 курс = 3 ролика = 90с → ~630 кредитов / ~$4.5 (без переделок).
  - **Буфер +20–30%** — проваленные генерации списывают кредиты без возврата.
- **Картинки:** Leonardo ~$0.01–0.02 за лист референса персонажа; обложки/декор/стиль-фото — копейки.
- **Озвучка:** ElevenLabs — центы за минуту; переозвучка точечно.
- **Инфра:** Supabase free / хостинг админки (Vercel/Netlify free) / email — $0. Tailscale не используется.

## 18. Что сознательно НЕ берём

- **`retelling`** (тип задания) — не нужен, в типах §6.5 спеки его нет.
- **`auto_check`** (поле) — убрано: все задания с проверкой требуют явного `Done` (§6.5а).
- **3 варианта сценария** — у нас итеративный единый канон, лучше.
- **`is_feed` / TikTok-лента внутри приложения** — вырезано целиком.
- **n8n / Make** — вся оркестрация на Python.
- **Midjourney** — картинки на Leonardo/Flux (C1a спеки); Midjourney не добавляем.
- **Tailscale / Jinja-HTMX / туннель** — апрувер в админке, связь через Supabase.
