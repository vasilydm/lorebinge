## A. Противоречия (схема расходится сама с собой)

### A-01 · `Streak.freezesAvailable` остался в доменной модели — 🔴 Critical

**Где:**
- Доменная модель: [31-app-screens-features.md:32-36](spec/31-app-screens-features.md#L32-L36) — `interface Streak { current; longest; freezesAvailable }`.
- Схема: [10-data-model.md:28-37](spec/10-data-model.md#L28-L37) — таблица `streaks` содержит только `current_streak`, `longest_streak`, `last_activity_date`; явно сказано «отдельного поля-счётчика заморозок в `streaks` нет».
- Дорожная карта: [34-app-roadmap.md:23](spec/34-app-roadmap.md#L23) — «`streaks` **без** `freezes_available`».
- RPC: [11-backend-api.md:19](spec/11-backend-api.md#L19) — `refresh_user_state` возвращает `frozen_days`, а не `freezes_available`.

**Проблема:** TypeScript-интерфейс `Streak` всё ещё описывает поле `freezesAvailable` — наследие старой механики ручной заморозки (магазин убран, КРИТ-8). Маппер DTO→модель не сможет его заполнить: в БД источника нет.

**Почему важно:** это единственное место во всём спеке, где доменная модель прямо противоречит схеме. Claude Code при генерации data-слоя (Phase 3) либо создаст «мёртвое» поле, либо упрётся в несоответствие zod-DTO ↔ модель. Высокий риск тихого бага или ступора генерации.

**Рекомендация:** убрать `freezesAvailable` из `interface Streak`. Если на HUD/в профиле нужна «валюта заморозки» — это и есть `diamonds` (§A5).

> **✍️ Ваш вердикт:**
> согласен

### A-02 · TTS-аудио играется через `expo-audio`/`expo-av`, которого нет в зависимостях — 🟠 Important

**Где:**
- Использование: [31-app-screens-features.md:466](spec/31-app-screens-features.md#L466) и [31-app-screens-features.md:505](spec/31-app-screens-features.md#L505) — «озвучка… стартует сразу (`expo-audio`/`expo-av`)», «Источник звуков — бандл-ассеты (`expo-audio`)».
- Зависимости: [33-app-implementation.md:17-53](spec/33-app-implementation.md#L17-L53) — в baseline `package.json` есть `expo-video`, но **нет** `expo-audio` и `expo-av`.
- Стек: [30-app-overview.md:23-54](spec/30-app-overview.md#L23-L54) — в таблице стека аудио-библиотека не упомянута (только видео — `expo-video`).

**Проблема:** TTS-озвучка сегментов в `reading_text`/`reading_media` (ключевая механика чтения, Phase 6) и звуки вердикта (§12.5г) требуют аудио-плеера, но он не зафиксирован в baseline. `expo-video` ≠ аудио-плеер; `expo-av` в новых SDK задепрекейчен в пользу `expo-audio`.

**Почему важно:** Phase 6 встанет без выбранной аудио-библиотеки; неопределённость `expo-audio` vs `expo-av` — ровно тот класс пробелов, который Прил. A призван закрывать (нужен пин под SDK 55).

**Рекомендация:** зафиксировать `expo-audio` в §A1 (baseline) и в таблице стека (§2), убрать упоминание `expo-av`. Заодно — это же аудио для эффект-звуков урока (§12.5г).

### A-03 · `UserProfile.isSubscriber`/`subscriptionBadge` — нет источника в `profiles` и нет RPC — 🟡 Minor

**Где:**
- Модель: [31-app-screens-features.md:12-23](spec/31-app-screens-features.md#L12-L23) — `UserProfile` включает `isSubscriber` и `subscriptionBadge`.
- Таблица `profiles`: [10-data-model.md:13-26](spec/10-data-model.md#L13-L26) — ни статуса подписки, ни бейджа в ней нет.
- Источник HUD: [31-app-screens-features.md:316](spec/31-app-screens-features.md#L316) — «прямое чтение **своих** строк `profiles` + `streaks`» (про `subscriptions` не сказано).

**Проблема:** чтобы собрать `UserProfile`, репозиторий должен склеить три источника: `profiles` + `subscriptions.status` (→ `isSubscriber`) + `app_config.subscription_badge` (→ `subscriptionBadge`). Это нигде не описано как контракт; §12.0 называет источником HUD только `profiles`+`streaks`.

**Почему важно:** HUD-слот «батарея ↔ ассет» зависит от статуса подписки. Если репозиторий не прочитает `subscriptions`, слот всегда будет показывать батарею. Контракт сборки `UserProfile` стоит проговорить (или ввести `get_me`-RPC, см. C-06).

**Рекомендация:** добавить в §12.0/§9 явную строку: `isSubscriber`/`subscriptionBadge` собираются из `subscriptions` + `app_config.subscription_badge`, инвалидируются вебхуком/`validate-purchase`.

### A-04 · Nullable-дрейф между моделью и БД — 🟡 Minor

**Где:**
- `Course.coverImageUrl: string` (обязателен): [31-app-screens-features.md:47](spec/31-app-screens-features.md#L47).
- `courses.cover_image_url text null`: [10-data-model.md:72](spec/10-data-model.md#L72).

**Проблема:** доменная модель объявляет поле непустым, в БД оно nullable. Аналогично `Lesson.cover_image_url` (null в БД). Маппер обязан коалесить в плейсхолдер, иначе zod-валидация DTO упадёт на реальной строке с `null`.

**Почему важно:** не блокер, но это источник рантайм-падений валидации на «живых» данных конвейера/админки, где обложка ещё не сгенерирована.

**Рекомендация:** либо сделать поле `string | null` в модели, либо явно прописать правило коалесинга `null → placeholder` в маппере (§9/A13 уже говорит про placeholder — связать явно).

## B. Отсутствующие критические блоки БД

### B-01 · Не задан ни один индекс — 🔴 Critical

**Где:** во всём §6 ([10-data-model.md](spec/10-data-model.md)) и Phase 1 ([34-app-roadmap.md:20-25](spec/34-app-roadmap.md#L20-L25)) нет ни одного `create index`. Единственная косвенная уникальность — инлайновые `text unique` на `username`, `courses.slug`, `categories.slug`.

**Проблема:** для server-authoritative приложения, где каждый запрос идёт через RPC по горячим путям, отсутствуют индексы под:
- `courses(is_published, sort_order)`, `courses(category_id)`, `courses(popularity_score)` — ленты Home/Grid/Catalog/Category;
- `lessons(course_id, sort_order)` — сетка уроков + «обложка первого урока»;
- `tasks(lesson_id, sort_order)` — раннер + агрегат `tasks_count`;
- `user_lesson_progress(user_id, lesson_id)` (PK покрывает) и `user_course_progress` (PK покрывает) — но `user_task_attempts(user_id, task_id)` индекса не имеет, а `complete_lesson` по нему джойнит;
- `diamond_transactions(user_id, reason, ref_id)` — проверка идемпотентности (см. B-02);
- `subscriptions(rc_app_user_id)`, `payment_events(user_id)` — вебхук резолвит юзера по `rc_app_user_id`.

**Почему важно:** без индексов на `ref_id`/`rc_app_user_id` идемпотентные проверки и резолв вебхука деградируют до seq-scan; ленты курсов будут медленными по мере роста каталога (а каталог растит конвейер). Это базовый блок DB-спека, которого сейчас нет.

**Рекомендация:** добавить в §6 (или в Phase 1) раздел «Индексы» с явным списком под горячие запросы RPC §8. Минимум — перечисленное выше.

> **✍️ Ваш вердикт:**

### B-02 · Идемпотентность алмазов без `UNIQUE`-констрейнта — 🔴 Critical

**Где:**
- [33-app-implementation.md:127](spec/33-app-implementation.md#L127) — «результат фиксируется записью в `diamond_transactions` (`reason='lesson_completion_reward'`, `ref_id=lesson_id`)»; «Взвешенный кубик разыгрывается один раз на урок».
- [11-backend-api.md:29](spec/11-backend-api.md#L29) — «начисление… идемпотентно (проверка по леджеру `ref_id`)».

**Проблема:** идемпотентность реализована как «проверь, нет ли уже строки в леджере, и если нет — вставь». Без `UNIQUE (user_id, reason, ref_id)` это **read-then-write гонка**: два параллельных `complete_lesson` (двойной тап, ретрай TanStack Query, дабл-сабмит) оба пройдут проверку «строки нет» и начислят алмазы дважды. Принцип §3 «идемпотентность операций» декларирован ([30-app-overview.md:78](spec/30-app-overview.md#L78)), но на уровне схемы не подкреплён.

**Почему важно:** это прямой путь к раздаче лишних алмазов (а алмазы = заморозка стрика = soft-валюта монетизации). «Анти-фарм не цель», но защита от **двойного начисления одной и той же награды** — это не анти-фарм, это корректность.

**Рекомендация:** добавить partial unique index: `create unique index on diamond_transactions(user_id, reason, ref_id) where reason in ('lesson_completion_reward','lesson_replay_reward')` (или с `ref_id is not null`), и делать `insert ... on conflict do nothing` с проверкой `rowcount`. Тогда идемпотентность гарантируется БД, а не порядком выполнения.

> **✍️ Ваш вердикт:**
> согласен
>

### B-03 · Не заданы правила `ON DELETE` для внешних ключей — 🔴 Critical

**Где:** во всех таблицах §6 FK описаны как «`uuid FK→profiles`» / «`uuid FK→courses`» без указания `ON DELETE`. `delete_account` ([11-backend-api.md:27](spec/11-backend-api.md#L27)) обещает «удаляет данные пользователя (**каскад**) и аккаунт».

**Проблема:** каскад обещан, но нигде не объявлен. Без `ON DELETE CASCADE` на пользовательских FK (`streaks`, `user_course_progress`, `user_lesson_progress`, `user_task_attempts`, `diamond_transactions`, `subscriptions`, `payment_events`) удаление аккаунта упрётся в нарушение FK. Аналогично контентные FK (`lessons→courses`, `tasks→lessons`, `media_assets→tasks`): при удалении курса в админке нужно решить `CASCADE` vs `RESTRICT`. Сейчас поведение не специфицировано → Postgres по умолчанию `NO ACTION`, и `delete_account`/удаление курса будут падать.

**Почему важно:** `delete_account` — **жёсткое требование обоих сторов** (Apple 5.1.1(v), Google). Если каскады не объявлены, фича не пройдёт ревью. Это критический и одновременно юридический блок.

**Рекомендация:** в §6 проставить `ON DELETE` для каждого FK: пользовательские → `CASCADE`; `payment_events.user_id` → `SET NULL` (аудит выручки переживает удаление аккаунта — иначе теряется финансовая история); `error_logs.user_id` → `SET NULL`. Контент (`lessons/tasks/media_assets`) → `CASCADE` от родителя.

> **✍️ Ваш вердикт:**
> согласен

### B-04 · Нет стратегии блокировок для мутаций `battery`/`diamonds` — 🟠 Important

**Где:**
- `submit_task_attempt`: [11-backend-api.md:16](spec/11-backend-api.md#L16) — «`battery -= battery_cost_per_answer`».
- Формула восстановления: [33-app-implementation.md:98](spec/33-app-implementation.md#L98) — read `last_battery_refill_at` → пересчёт → write.

**Проблема:** списание батареи и алмазов описано как «прочитал значение → посчитал → записал». Параллельные вызовы (быстрые тапы, ретраи) могут прочитать одно и то же значение и затереть друг друга (lost update). Для SECURITY DEFINER RPC нужна явная стратегия: `SELECT … FOR UPDATE` на строке `profiles` или атомарный `UPDATE … SET battery = greatest(0, battery - cost) WHERE … RETURNING`.

**Почему важно:** экономика заявлена server-authoritative, но без блокировки гонка позволяет ответить дважды «за один заряд» или уйти в неконсистентный баланс. Тонкий, но реальный конкурентный баг.

**Рекомендация:** в §8.1 добавить инвариант: все мутации `battery`/`diamonds`/`current_task_index` выполняются под `FOR UPDATE` строки пользователя (или атомарным UPDATE с условием). Один абзац-правило закроет весь класс.

> **✍️ Ваш вердикт:**
> согласен

### B-05 · Нет ключа идемпотентности у `payment_events` — 🟠 Important

**Где:** [10-data-model.md:187-202](spec/10-data-model.md#L187-L202) (таблица `payment_events`) и [11-backend-api.md:37](spec/11-backend-api.md#L37) (`revenuecat-webhook`).

**Проблема:** вебхуки RevenueCat доставляются **at-least-once** (возможны повторы/ретраи). У `payment_events` нет уникального ключа события (напр. RevenueCat `event.id`). Повторная доставка `initial_purchase`/`trial_started` → дубль строки аудита и потенциально повторное применение сайд-эффекта (`trial_used=true` идемпотентно, но `renewal`/счётчики — нет).

**Почему важно:** дубли искажают аналитику выручки в админке и могут повторно тронуть `subscriptions`. Стандартная защита вебхуков отсутствует.

**Рекомендация:** добавить `event_id text unique` (из payload RevenueCat) в `payment_events`; вебхук пишет `on conflict (event_id) do nothing` и при конфликте выходит рано.

> **✍️ Ваш вердикт:**

## C. Пробелы в backend-логике

### C-01 · Кто возвращает батарею при истечении подписки? — 🟠 Important

**Где:** [33-app-implementation.md:104](spec/33-app-implementation.md#L104) — «при отмене/истечении… батарея возвращается: `battery = battery_max`, `last_battery_refill_at = now()`».

**Проблема:** описан результат, но не **функция-исполнитель**. Кто делает этот переход — `revenuecat-webhook` на событии `expiration`/`cancellation`? Или лениво `refresh_user_state` при следующем заходе? Если только вебхук — у юзера без сети/без события батарея не вернётся; если только refresh — то до первого захода статус рассинхронизирован.

**Почему важно:** «возврат батареи» — это точка монетизации наоборот (бывший подписчик снова в лимитах). Неопределённость = либо двойной сброс (и вебхук, и refresh), либо ни одного.

**Рекомендация:** зафиксировать: сброс делает `refresh_user_state` лениво при обнаружении перехода `active/trial → expired/cancelled/none` (одно место, переживает отсутствие вебхука). Вебхук только обновляет `subscriptions.status`.

### C-02 · Проверка «все задания пройдены» ломается на перепрохождении — 🟠 Important

**Где:**
- [33-app-implementation.md:141](spec/33-app-implementation.md#L141) — «`complete_lesson` проверяет, что для всех `tasks` урока есть хотя бы одна запись в `user_task_attempts`».
- [33-app-implementation.md:143-144](spec/33-app-implementation.md#L143-L144) — при перепрохождении `current_task_index`/`errors_count` сбрасываются в 0, но про удаление `user_task_attempts` ничего не сказано.

**Проблема:** `user_task_attempts` — это накопительная история (id, user_id, task_id, is_correct, attempted_at), без удаления при replay. Значит после первого прохождения для всех задач уже есть ≥1 attempt **навсегда**. На перепрохождении серверная проверка «у всех задач есть attempt» проходит мгновенно, ещё до того как пользователь реально ответил заново. Серверный guard завершения становится бессмысленным для replay — фактическое завершение целиком на совести клиента.

**Почему важно:** спек честно говорит «момент завершения контролирует клиент (осознанное упрощение)» — но тогда проверка attempts в `complete_lesson` создаёт ложное ощущение серверной защиты, которой на replay нет. Либо guard надо делать честным, либо признать, что его нет.

**Рекомендация:** на выбор — (а) при `start_lesson` для replay удалять `user_task_attempts` урока (тогда guard снова работает), либо (б) проверять завершение по «`current_task_index` дошёл до конца», а не по наличию attempts. Вариант (б) проще и согласован с `current_task_index` как единственным курсором.

> **✍️ Ваш вердикт:**
> да прогресс есть уже после первого прохождения и перепрохождение нужно посто так - если пользоатель хочет - ни на что не влияет

### C-05 · `payment_events.user_id` — FK без `null`, но резолв по `rc_app_user_id` — 🟡 Minor
